01

設計理念

Why it works

產線的四根柱子。每一根都在回答同一個問題：當 AI 寫程式的量遠超過人 review 的量，錯誤靠什麼被擋下來？

PILLAR A

確定性控制

流程順序 = 你逐一下的 /feat-* 指令；寫入範圍 = scope-track.sh 依 .active 硬擋；迴圈 = retries 計數器在 loopMaxRetries 停損。凡是能用確定性程式碼保證的，絕不交給模型判斷（CLAUDE.md Rule 9）。

PILLAR B

檔案交接

每個 subagent 都在全新、隔離的 context 啟動，唯一共享的是檔案系統。story.md → brief.md → backend-summary.md → verification.md——交接物便宜、可 review、可續跑，也讓每段工作的 context 小到可以放進最便宜夠用的模型。

PILLAR C

獨立驗證

「自己改的考卷不算數。」validator 沒有 Edit/Bash 工具、只能讀和寫報告；test-verifier 從 story 的驗收標準獨立出題。評估自己輸出的模型會偏袒自己先前的推理——所以驗證者永遠是另一個 context。

PILLAR D

Fail-Loud 誠實

每個 agent 的每次回報都以三段結尾：✅ Verified／⚠️ Skipped-Uncertain／❓ Needs-human-input。「tests pass」永遠不准掩蓋被跳過的測試。寧可對不確定大聲，不要對交付安靜。

02

產線圖

The pipeline

一個 feature 從進站到出站。⏸ 黃色閘門 是人工核准，產線在此完全停止；FABLE5 紫色標記是本次升級新增的站點，其餘與原版相同。每站的產出都是 <artifactsDir>/<slug>/ 下的一個檔案。

/feat-init→ project.json（每 repo 一次）

偵測技術棧與 tracks（backend／frontend），寫出 manifest。全新 repo 讀不到 build 檔時會直接問你。只寫設定，不 scaffold 程式碼。

/feat-new <slug> "描述"→ idea.md + state.json

開一個 feature 的工作資料夾，記下原始需求。

/feat-researchFABLE5 強化→ research.md

先讀 MEMORY.md 把過往教訓帶成「Prior knowledge」（是待驗證的線索，不是可信的事實），再用唯讀 Explore 掃碼庫：相關檔案、既有模式、風險。

/feat-story⏸ 人工核准→ story.md

story-writer 產出使用者故事＋可測試的驗收標準。你必須親自看過並核准，產線才會繼續——這是第一道判斷閘門。

/feat-spec⏸ 人工核准→ brief.md

spec-writer 把已核准的故事轉成技術簡報：資料模型、API 契約、要動的檔案、風險。第二道判斷閘門。

/feat-backend→ 程式碼 + backend-summary.md

backend-builder 依 brief 實作＋單元測試，並寫出 API 契約摘要給前端。hook 把它鎖在 backend 目錄內。（無 backend track 則自動跳過）

/feat-frontend→ UI + 元件測試

frontend-builder 依契約實作。契約缺什麼就回報缺口，絕不自己發明 endpoint。

/feat-shipFABLE5 新增選用搭配 /goal

把 06→08 的機械循環交給 /goal 自動收斂：指令會印出一條精確的 goal 條件讓你貼上，獨立評估者（預設 Haiku）逐回合判定是否達標。不想用就照舊手動跑 06→08。

/feat-verify→ verification.md

test-verifier 從 story 的每條驗收標準寫驗收測試並執行，逐條記 pass／fail。寫不出會失敗的測試 = 不准寫。

/feat-validate→ validation.md

validator 唯讀審查：驗收、安全、範圍、慣例。發現分級 🔴 Critical／🟠 Important／⚪ Minor，每條附 path:line。乾淨 → 解除 .active，下一站 /feat-docs 產出使用者文件。

/feat-fix有界迴圈retries ≤ loopMaxRetries(3)

把 🔴🟠 與紅測試派回所屬 track 的 builder 外科手術式修復，然後回到 06 重驗。觸頂即標 blocked 停手交人——迴圈永遠不會失控。

↺ fix 完成後回流 /feat-verify → /feat-validate；達 3 次上限 → blocked，交人工判斷
/feat-docs→ README + docs/（EN → zh-TW）

doc-writer 依「實際做出來的東西」撰寫使用者文件：更新 README、在 docsDir 下產出使用／範例指南（含 Mermaid 圖），先英文、再 _zh-TW 翻譯，兩版頂部皆有語言切換連結。hook 限制它只能寫 docsDir 與根目錄 README。

/feat-distillFABLE5 新增→ MEMORY.md 更新

memory-distiller 把這個 feature「實際發生過」的教訓蒸餾進跨 feature 的 MEMORY.md（Verified facts／General rules／Watchlist）。失敗也要蒸餾：blocked 的開放問題進 Watchlist，下個 feature 不重踩。零新教訓是合法結果，禁止硬湊。

隨時可用 /feat-status <slug> 查詢任何 feature 在產線上的位置與下一步。

03

使用技術

Under the hood
機制做什麼關鍵事實
Subagents.claude/agents/*.md（8 個） 每個角色一個 agent 檔：frontmatter 宣告工具與模型，body 是它的系統提示。啟動時 context 全新隔離，只透過檔案交接。 工具即權限——validator 沒有 Edit/Bash，想改也改不了。
Model 路由 FABLE5frontmatter · model: sonnet 編排 session 用貴的模型（Fable 5），worker 全數釘在 Sonnet。切換 subagent 模型不會重讀 context——本來就從零開始。 同樣的檔案讀取量，用 Sonnet 費率計價；主 session 的 cache 不受影響。
Hooks.claude/hooks/*.sh（3 個） scope-track.sh 依 .active 硬擋越界寫入；protect-secrets.sh 擋 .env／金鑰的寫入與 commit；quality-gate.sh 依 manifest 跑 typecheck→lint→test。 需要 jq；缺 jq 時 hooks fail-safe 放行並提示（＝強制沒開）。無 .active 時日常編輯永不受限。
Manifest.claude/factory/project.json 每 repo 一份：tracks、目錄、指令、protectedDirs、loopMaxRetries。所有指令／agent／hook 都讀它——同一套 template 通吃全端、純前端、純後端。 由 /feat-init 產生；沒有的 script（如 lint）留空，quality-gate 自動跳過。
/goal 收斂 FABLE5Claude Code 內建 · /feat-ship 配套 由使用者設定目標條件；每回合結束後獨立評估者（預設 Haiku）判定達標與否，未達標帶理由續跑。 評估者只看對話，不碰檔案系統——所以每輪都必須貼出證據。/goal 屬 hooks 系統：disableAllHooks 會連 scope 強制一起殺掉，絕不在 hooks 停用下跑產線。
記憶層 FABLE5.claude/factory/MEMORY.md 跨 feature 的長期記憶，只由 memory-distiller 寫入、由 /feat-research 開場讀取。條目必須可追溯到實際發生的事，硬上限 ~150 行。 放在 repo 裡、進版本控制——記憶的 diff 要像程式碼一樣 review，壞記憶的複利跟好記憶一樣快。
04

Fable 5 升級了什麼

The upgrade

四件事，全部以 FABLE5 標記散落在檔案裡，git diff 一眼可見。原則：演進，不重建——兩道 ⏸ 閘門、hook 硬擋、有界迴圈一個都沒動。

① Model 路由

編排者與工人分開計價：

角色模型
編排 session（你操作的）Fable 5
全部 8 個 agentssonnet（frontmatter 釘死）
/goal 評估者Haiku（內建預設）
安全敏感工作該 agent 改 model: opus

② Classifier 拒絕 ≠ 缺陷

Fable 5 內建安全分類器可能拒絕請求（API 回 stop_reason: "refusal"）。分不清「分類器說不」和「程式碼有錯」的迴圈，會把 retry 額度燒在換句話說上。因此：

  • agents 遇拒絕記 classifier-refusal，禁止改寫繞過
  • validator 另立 ⛔ 分類（owner: human）
  • /feat-fix 第 0 步分流：全是拒絕 → blocked-classifier，不消耗 retry

③ 記憶層

沒有它，每個 feature 的教訓都隨 artifacts 過期而蒸發。/feat-distill 在驗證通過後（或 blocked 時）把教訓入庫；/feat-research 開場查閱。產線從「每次歸零」變成「續跑」——這是文章說的「複利」真正落地的地方。

④ /goal 收斂迴圈

/feat-ship 讓 verify→validate→fix 自動收斂，goal 條件內建三個硬限制：

  • 每項達標條件都要求「對話中實際展示的輸出」為證
  • 條件自帶回合上限（軟）；loopMaxRetries 仍是硬停損
  • blocked／blocked-classifier／⛔ 明文定義為「無法達標」，大聲終止而非空轉
惰性無害設計：這些升級在其他模型（Opus、Sonnet）的 session 下照常可用或自然閒置——model 欄位是一般路由（省錢）、記憶與 /goal 模型無關、classifier 路徑在非 Fable 模型下永不觸發。所以全團隊只維護這一套 template，模型用 /model 選，不用換 template 選。

參考來源：本次升級的設計靈感來自 @0xCodez 的 self-improving agent 討論串（動區 BlockTempo 中譯版）。文章屬靈感而非規格——所有產品行為主張均已對照 Anthropic 官方文件查證後才採用。

使用方法

How to run it

安裝（每個 repo 一次）

# 1. 從升級版 template 複製進你的 repo ./install.sh /path/to/your/repo # Windows：./install.ps1 C:\path\to\repo（用 Git Bash/WSL 跑 Claude Code） # 2. 確認環境：bash + jq 在 PATH 上 jq --version # 3. 在 repo 開 Claude Code，初始化 manifest /feat-init # 偵測技術棧或直接問你；只寫 project.json # 4. 已有 CLAUDE.md → 合併兩個 <<<FEATURE FACTORY>>> 區塊 # 全新 repo → snippet 已自動成為起始 CLAUDE.md，免合併 # 5. 把 .claude/ + CLAUDE.md 一起 commit

跑一個 feature（標準路徑）

/feat-new user-invite "邀請成員加入工作區" /feat-research user-invite /feat-story user-invite # ⏸ 看過 story.md 再往下 /feat-spec user-invite # ⏸ 看過 brief.md 再往下 /feat-backend user-invite # 有 backend track 才有這步 /feat-frontend user-invite # A 路線：手動收斂 /feat-verify user-invite → /feat-validate user-invite # 有 findings → /feat-fix，再回 verify/validate # B 路線（FABLE5）：自動收斂 /feat-ship user-invite # 照它印出的 /goal 貼上執行 # 收尾：使用者文件 → 教訓入庫 → 自己 review + commit /feat-docs user-invite # README + docs/ 指南（英文→繁中，含 Mermaid） /feat-distill user-invite # 產線不代你 commit、不開 PR

狀態對照：卡住時看這裡

你看到的意思／怎麼辦
validate 乾淨跑 /feat-docs 產出使用者文件，收尾 /feat-distill，然後自己 review、自己 commit。
blockedfix 迴圈達 3 次上限，需要人的判斷。先 /feat-distill 把開放問題入 Watchlist，再人工處理。
blocked-classifier被安全分類器拒絕，不是程式碼壞了。把該步驟的 agent 換 model: opus 重跑，或改人工處理。
寫入被 hook 擋下設計如此——當前 track 不准碰那個路徑。不要繞過；真有需要，rm .claude/factory/.active 解除（等於離開產線模式）。
/usage 中 Sonnet 掛零路由被 env 變數覆蓋(任何層級的 CLAUDE_CODE_SUBAGENT_MODEL 都會壓過 frontmatter)。查 echo $CLAUDE_CODE_SUBAGENT_MODEL;詳見 CONVENTIONS「Known issues」。
「jq not found」強制機制其實沒開（fail-safe 放行）。裝好 jq 再繼續。
quality-gate 顯示 (none configured, skipped)manifest 裡該指令留空。專案補上 test/lint script 後回填 project.json。
06

團隊守則

House rules
產線 = Default Mode

進工廠的是「決定要出貨」的 feature。prototype／spike／POC 走 EXPLORE Mode，不進 /feat-*——所以測試紀律永遠不會跟探索打架。

閘門是給人守的

⏸ story 和 spec 一定要真的看。哪天你發現自己每次都不看直接核准——那是重新設計節拍的訊號，不是省時間的技巧。

MEMORY.md 是程式碼

它進版本控制、它的 diff 要 review。壞記憶會被每個後續 feature 讀到——複利是雙向的。

不繞過、不美化

hook 擋你是刻意的；「tests pass」不准掩蓋跳過的測試；goal 靠證據達標，不靠敘述達標。對不確定大聲，是這條產線唯一的信仰。
